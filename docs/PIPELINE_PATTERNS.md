# Pipeline Patterns

Pipeline authors face a choice at every node: should this be an LLM call (`shape=box`,
codergen handler) or a direct tool call (`shape=parallelogram`, tool handler)? The
strongdm/attractor nlspec defines the mechanisms for both node types but does not articulate
when to use which. This document captures the layering discipline that effective attractor
pipelines practice — and names it so pipeline authors can apply it deliberately.

**Layering:** Pipeline authoring discipline sits above the engine layer. The engine executes
whatever graph it is given; it does not enforce this discipline. These patterns are for
pipeline authors and pipeline reviewers, not the engine itself.

**Relationship to other docs:** Engine-level contracts (M5 substitution, fail-fast policy,
structural concurrency) are in `docs/CONTRACTS.md`. Node shape syntax, edge conditions, and
handler attributes are in `docs/DOT-AUTHORING-GUIDE.md` and `docs/DOT-SYNTAX.md`. This
document covers design discipline only.

---

## 1. The Layering Principle

> **Use code where models are unreliable or inconsistent. Use models where code is hard or
> impossible. Never use a model as a format translator that only exists to produce input for
> the next code step.**

**What makes models unreliable at a task:**

- Producing exact-format output (unified diffs, structured manifests, routing sentinels) —
  variance increases with model age and is higher on local or smaller models
- Deterministic computation (math, counting, file manipulation, state management)
- Anything `grep`, `jq`, or `bash` can do faster and without variance

**What makes code insufficient:**

- Judgment under ambiguity (analysis, design review, synthesis, planning)
- Content generation (writing, coding, summarization)
- Multi-step reasoning across complex, unstructured context

The quick self-test: **"Is the model here for judgment, or just to type?"** If the answer is
"just to type," replace it with a parallelogram node.

---

## 2. The Right-Layer Test

Apply this test at design time before committing a node to a particular shape:

> If you removed the `agent → format artifact → code mechanically processes format` chain
> and had the agent do the end goal directly with file tools, would anything functionally be
> lost?

| Answer | What it tells you | Strategy |
|---|---|---|
| Nothing is lost — the format is plumbing | The model's actual goal is the effect (edits, state changes). The format is accidental. | SF: Skip the Format (§3) |
| The format IS the deliverable — it is consumed by humans or stored as an artifact | The format carries semantic value. Drop the format and you lose the output. | V+R: Validation + Retry (§4) |
| The entire node is deterministic | No judgment required at all. | Code-only (§5) |

---

## 3. Strategy SF — Skip the Format

**When:** The structured format only exists so the next code step can process it. The real
goal is the effect — file edits, state changes, data mutations.

**Pattern:**

```
agent (does actual work with file tools)
  → code (captures result + validates with deterministic primitives)
```

The agent uses its native file-editing tools to do the work directly. A downstream
parallelogram node captures the result (for example, via `git diff`) and a second
parallelogram runs code-level validation. Neither the diff format nor the routing signal is
generated by an LLM at any point.

**DOT skeleton:**

```dot
EditWork [
    shape=box,
    prompt="Use your file tools to make the changes described in $goal.
            Your edits will be validated by code after this node."
];

CaptureResult [
    shape=parallelogram,
    tool_command="cd $work_path && git diff HEAD > result.diff
                  && printf 'diff_captured' || printf 'no_changes'"
];

ValidateResult [
    shape=parallelogram,
    tool_command="python -m myproject.validate --path $work_path
                  && printf 'valid' || printf 'invalid: see output above'"
];

EditWork      -> CaptureResult;
CaptureResult -> ValidateResult [condition="context.tool.last_line=diff_captured"];
CaptureResult -> Done           [condition="context.tool.last_line=no_changes"];
ValidateResult -> Continue      [condition="context.tool.last_line=valid"];
ValidateResult -> Report        [condition="context.tool.last_line=invalid"];
```

**Why it works:** `git diff` and the validator are deterministic. Path-format variance
(a known LLM failure mode when generating unified diffs: with vs. without directory prefix,
hunk context line counts) is structurally eliminated — git produces the diff, not the model.
The validator's exit code is the routing signal, not any LLM output.

**When SF fits:** unified diffs, JSON manifests, custom-delimited code blocks parsed by
regex, XML change descriptions — any intermediate format that only exists to feed the next
code step.

**In-the-wild adoption.** Pipeline authors in the broader DOT ecosystem have independently
converged on Strategy SF for use cases like language-to-language code porting. A
representative third-party pattern: the implementation node uses file tools to write changes
directly across target files; a downstream build gate (compiler or test runner) validates via
deterministic exit code — no intermediate format to specify, no format variance to absorb.

The within-ecosystem contrast is instructive: earlier iterations of the same work used the
LLM-as-format-translator pattern (model produces structured code blocks → code parses →
writes files), producing silent file drops when block formatting varied across model runs.
Later iterations by the same author group adopted Strategy SF — from reliability pressure,
not from this document. That independent convergence is corroborating evidence.

## 4. Strategy V+R — Validation + Retry

**When:** The structured output IS the deliverable — it is consumed by humans, stored as a
project artifact (plan document, design brief, structured report), or carries semantic value
that cannot be replaced by a plain file edit.

**Pattern:**

```
agent (writes structured output)
  → code (validates structure)
    → if valid → continue
    → if invalid → loop back to agent with error context
```

Self-correcting in 1–2 iterations. The validator's feedback is precise and the retry rarely
fails twice, because the agent receives an exact error message rather than a vague "try again."

**DOT skeleton:**

```dot
DraftOutput [
    shape=box,
    prompt="Write a plan with exactly these sections: Goals, Steps, Risks.
            Return only the plan — no preamble or explanation."
];

ValidateOutput [
    shape=parallelogram,
    tool_command="grep -q '## Goals' output.md
                  && grep -q '## Steps' output.md
                  && grep -q '## Risks' output.md
                  && printf 'format_ok'
                  || printf 'format_fail: one or more required sections missing'"
];

DraftOutput   -> ValidateOutput;
ValidateOutput -> NextStep    [condition="context.tool.last_line=format_ok"];
ValidateOutput -> DraftOutput [condition="context.tool.last_line~format_fail"];
```

**When V+R fits:** design briefs, plan documents, structured reports for humans, config
templates that follow a required schema — anywhere the format itself carries value
independent of any downstream code step.

**Key detail:** the agent's prompt should state that its work will be validated by code
(so the agent knows to be careful), but need not enumerate the exact validation checks
inline. See Anti-Pattern AP-3 (§6) for the drift trap that specific enumeration creates.

---

## 5. Code-Only Is Sometimes the Right Answer

Some nodes require no judgment. Don't add LLM calls where parallelogram nodes suffice.

**When code-only fits:**

| Task type | Examples |
|---|---|
| State management | Sprint counters, ledger updates, flag toggles, context population |
| Pure transformation | Format conversion, data extraction, file copies, archive creation |
| Deterministic gates | Build/test runs, file presence checks, schema validation, exit-code routing |
| Output capture | `git diff`, `wc -l`, grep counts, checksum computation |

**Entire code-only pipelines are legitimate.** All-parallelogram pipelines built from
`grep`, `bash`, `jq`, and file-write operations are valid attractor graphs. The engine
executes them without modification. See `examples/pipelines/` for structural examples;
the parallelogram-heavy portions of those pipelines demonstrate this pattern.

## 6. Anti-Pattern Catalog

### AP-1: LLM as Format Translator

```dot
// PROBLEM: model only exists to produce format for the next code step
ProduceDiff [shape=box, prompt="Write a unified diff in exactly this format: ..."];
ApplyDiff   [shape=parallelogram, tool_command="git apply diff.patch"];
```

**Why it fails:** unified diff path format has LLM-generated variance (with vs. without
directory prefix, inconsistent context line counts, wrong hunk headers). Local and smaller
models fail more often. The model's only job here is to type something that code can
produce deterministically.

**Fix:** Use Strategy SF (§3). Have the model make the actual edit with its file tools;
capture the diff with `git diff HEAD`. Git produces it; the LLM does not.

---

### AP-2: LLM-Emitted Routing Sentinel

```dot
// PROBLEM: routing depends on exact LLM output
DoWork [shape=box, prompt="At the end, output exactly the word 'success' or 'failure'."];
DoWork -> Continue [condition="context.tool.last_line=success"];
DoWork -> Retry    [condition="context.tool.last_line=failure"];
```

**Why it fails:** LLMs are inconsistent at exact single-token output:

- Local and smaller models add commentary, mis-case, or paraphrase
- Newer frontier models are more reliable but not invariant
- Pipelines intended to be portable across models (including local models) cannot rely on
  exact LLM-emitted sentinels for routing without accepting occasional silent misfires

**The robust pattern:** have the agent write natural output to a file; a parallelogram node
then greps for keywords and emits the routing signal via `printf`. The agent is never
responsible for producing the exact sentinel text.

```dot
// FIX: code emits the routing signal
DoWork       [shape=box,
              prompt="Analyze X. Write your verdict to verdict.txt."];
CheckVerdict [shape=parallelogram,
              tool_command="grep -qi 'approve' verdict.txt
                            && printf 'approved' || printf 'rejected'"];

DoWork       -> CheckVerdict;
CheckVerdict -> Continue [condition="context.tool.last_line=approved"];
CheckVerdict -> Retry    [condition="context.tool.last_line=rejected"];
```

`printf` is deterministic. The routing signal comes from `grep`'s judgment of the agent's
natural output — not from the agent typing an exact keyword.

**Note on `goal_gate` / `outcome=success/fail/retry`:** The attractor engine provides
`goal_gate=true` as the spec-designed mechanism for outcome-based routing. When set, the
engine interprets the LLM's response to compute the node's `outcome` status, which
conditional edges route on (`condition="outcome=success"`, `condition="outcome=fail"`, etc.).
Pipeline authors using `goal_gate` are not violating spec — this is the designed-in path for
verdict routing.

However, `goal_gate` shares the underlying fragility of LLM-emitted routing: the engine's
interpretation depends on the LLM producing the expected outcome signal. Local models,
smaller frontier models, and models with verbose tendencies can fail this silently.

**Default lean:** if the LLM is already writing evidence to a file as part of its work
product, route on that file (LLM writes → parallelogram greps → routes via `printf`). This
produces the same outcome with less variance and no reliance on exact LLM token output.

**When `goal_gate` is the right choice:**

- The routing decision depends purely on LLM judgment with no file artifact
- The pipeline targets only frontier models in practice
- Adding a file-grep parallelogram would be ceremony with no real value

**When to prefer file-based routing over `goal_gate`:**

- The LLM is already writing evidence to a file (audit log, review verdict, validation result)
- Local model portability is a goal
- The pipeline runs across model families where exact-token output varies

Community canonical pipelines use `goal_gate` extensively for verdict nodes and that works in
practice for their target environments. Newer pipeline patterns favor file-based routing when
the evidence file already exists for other reasons. This is a design trade-off — not a
hierarchy of right and wrong.

### AP-3: Prompt/Validator Drift

```dot
// RISKY: validation specifics written in two places
DoWork   [shape=box, prompt="Output will be checked for:
           (1) JSON with a 'name' key, (2) 'status' field set to active or inactive..."];
Validate [shape=parallelogram, tool_command="python check.py"];
```

**Why it fails:** when `check.py` changes, the prompt becomes stale. Agents receive
misleading guidance, fail on checks they were not warned about, and produce confusing
diagnostics. This is a documentation drift trap, not a code bug.

**Two clean options — choose based on whether the agent needs specifics:**

1. **Generalize the notice:** "Your work will be validated by code to ensure structural
   integrity." No specifics in the prompt; no drift surface. Use this when the agent's
   natural caution covers it.

2. **DRY at runtime:** expose a `describe_checks()` method on the validator; a setup node
   writes its output to a context file (e.g., `validation_contract.md`) at pipeline start;
   the agent's prompt references the file. Single source of truth in code; both agent and
   human read from the same place.

Never enumerate validation specifics as a literal string in a prompt alongside a separate
validator that implements the same checks. That is the drift trap.

## 7. Routing Discipline After Agent Work

When an agent performs work and a downstream node needs to route on the result, three
approaches ranked by robustness:

| Approach | Robustness | Recommendation |
|---|---|---|
| LLM emits exact routing keyword as final output | Low | Avoid for portable pipelines (see AP-2) |
| LLM writes natural output → parallelogram greps → `printf` sentinel | Medium | Acceptable when agent output is a natural artifact that grep can interrogate |
| Agent does actual work (file edits, state changes) → code observes observable result (exit codes, `git diff`, file presence, validator output) | High | Prefer — Strategy SF naturally produces this |

The observable-result pattern is the most robust because neither the routing signal nor any
intermediate format is generated by an LLM at any point. Code observes a concrete change
in the world and routes from that observation.

**One-paragraph summary:** design nodes so that routing signals come from things the engine
or code can observe deterministically — exit codes, file presence, `git diff` output,
validator return codes. When agents must make decisions that affect routing, have them write
natural output and let code interpret it with grep or jq, not by matching LLM-typed keywords
verbatim.
